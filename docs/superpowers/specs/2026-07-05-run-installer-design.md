# Self-extracting `.run` GUI installer

**Date:** 2026-07-05
**Status:** Approved

## Problem

Everything in this project — the C daemon, the toggle script, the systemd
unit, the desktop file, the global shortcut binding — was set up by hand on
the author's own machine. `SETUP.md` is a note to self, not an installer.
There is no way for anyone else to install this and bind it to a key of
their choosing without reading the source and repeating each manual step.

## Constraints (inherent to the project, not this feature)

This daemon only works on **KDE Plasma on Wayland with KWin**. It uses:
- `zwlr_layer_shell_v1`, `zwp_idle_inhibit_manager_v1`,
  `zwp_keyboard_shortcuts_inhibit_manager_v1`, `zwp_pointer_constraints_v1`
  (KWin-specific Wayland protocol support)
- `org.kde.touchpad` and KDE PowerDevil D-Bus services
- `kglobalaccel` (KDE's global shortcut system) via a `.desktop` file

It cannot run on GNOME/Mutter, other Wayland compositors, or X11 sessions.
The installer's job is to make this constraint fail loudly and specifically,
not to work around it.

**Audience:** treated as small-scale/public sharing (a handful of users or a
low-traffic public post), not a hardened mass-market release. Pre-flight
checks give clear, specific error messages, but the installer does not need
to gracefully handle every conceivable misconfiguration. (This assumption
was not confirmed live with the user — flagged for their review.)

## Decision

A single self-extracting shell `.run` file: a small stub script with a
`tar.gz` of the packaged project appended to it. Running it extracts to a
temp directory and runs `install.sh`, which drives the whole install through
native `kdialog` dialogs (already present on every KDE system — zero added
runtime dependency) and builds the daemon **from source** on the target
machine rather than bundling a prebuilt binary, since Wayland client library
versions and available protocol extensions vary across distros/KWin
versions in ways a single prebuilt binary can't safely span.

**Key binding is not generated by the installer.** KDE serializes some keys
in `kglobalshortcutsrc` as named strings rather than plain keysyms — this
project's own F12 binding is stored as `Launch (1)`, not a keysym — so
hand-generating shortcut syntax for an arbitrary user-chosen key is fragile.
Instead, the installer registers the `.desktop` file (making it visible to
`kglobalaccel`) with no shortcut bound, then opens KDE's own Shortcuts
settings page (`kcmshell6 kcm_keys` / `systemsettings6 kcm_keys`) so the user
presses whatever key they want using KDE's native capture, which already
handles every key-naming edge case correctly. This is the mechanism that
actually delivers "bind it to any key," not a limitation being worked around.

## Alternatives considered

- **Bundle a prebuilt binary instead of building from source.** Faster
  install, no build-tool dependency on the target machine — but no CI
  matrix exists to validate a single binary across distros/Wayland versions,
  and static linking has its own portability caveats. Rejected: source build
  is the honest choice for a project with one contributor and one test
  machine.
- **A compiled Qt/GTK GUI installer app instead of a `kdialog` wizard.**
  Nicer visuals, but adds a whole toolkit dependency and reinvents
  key-capture that KDE's Shortcuts KCM already does correctly. Rejected as
  disproportionate effort for a personal utility's installer.

## Components

### Build side — `make dist` (new Makefile target, top-level or in a new `packaging/` dir)

Packages into a `tar.gz`:
- `src/` (the daemon source + Makefile — built on the target machine, not here)
- `systemd/blackout-overlay.service`
- `bin/blackout-toggle`
- generated `install.sh` and `uninstall.sh` (below)

Appends the `tar.gz` to a shell stub to produce `blackout-screen-install.run`.
The stub: extracts the appended archive to a `mktemp -d` directory, `cd`s
into it, execs `install.sh`, then cleans up the temp directory on exit
(success or failure).

### `install.sh`

1. **Pre-flight checks**, each a `kdialog --error` + exit on failure:
   - `$XDG_SESSION_TYPE = wayland` and `pgrep -x kwin_wayland` — confirms
     KDE Plasma on Wayland. Error message names exactly what was expected
     ("This tool requires KDE Plasma running on Wayland (KWin). Your
     session is: $XDG_SESSION_TYPE / $XDG_CURRENT_DESKTOP").
   - `command -v gcc pkg-config wayland-scanner` and
     `pkg-config --exists wayland-client` — confirms build tooling. Error
     names the missing package(s) generically (exact package names vary by
     distro; the message says "install your distro's wayland-client
     development headers and a C compiler" rather than guessing a specific
     `dnf`/`apt` invocation).
   - `command -v busctl kdialog systemctl` — confirms the runtime tools the
     daemon and this installer both depend on.
2. **Build:** `make -C src/blackout-overlay-c`. On failure, `kdialog --error`
   showing the last ~20 lines of build output, then exit.
3. **Install:**
   - `install -Dm755 src/blackout-overlay-c/blackout-overlay ~/.local/bin/blackout-overlay`
   - `install -Dm755 bin/blackout-toggle ~/.local/bin/blackout-toggle`
   - `install -Dm644 systemd/blackout-overlay.service ~/.config/systemd/user/blackout-overlay.service`
   - Generate `~/.local/share/applications/net.local.blackout-toggle.desktop`
     with `Exec=%h/.local/bin/blackout-toggle` and no
     `X-KDE-Shortcuts=` line (unbound by default).
   - Copy `uninstall.sh` (with the install-time paths already substituted in)
     to `~/.local/share/blackout-screen/uninstall.sh`.
4. **Activate:** `systemctl --user daemon-reload && systemctl --user enable --now blackout-overlay`.
5. **Final dialog:** `kdialog --msgbox` explaining the daemon is running and
   the next step is choosing a key, then launch
   `systemsettings6 kcm_keys` (fall back to `kcmshell6 kcm_keys` if the
   former isn't found) so the user can search "blackout-toggle" and press
   their chosen key.

### `uninstall.sh`

Stops and disables the systemd service, removes the four installed files
(`blackout-overlay`, `blackout-toggle`, the `.service` file, the `.desktop`
file), and shows a `kdialog --msgbox` reminding the user that the shortcut
binding itself (if they set one) must be removed via the same Shortcuts
page — there's no clean way to un-bind a `kglobalshortcutsrc` entry from a
script without KDE's own tooling, matching the installer's own philosophy
of not hand-editing that file.

## Error handling

- Every pre-flight check and the build step fail through the same pattern:
  specific `kdialog --error` message naming what's wrong, then `exit 1`.
  No partial installs — if the build fails, nothing has been installed yet
  (install happens only after a successful build).
- `systemctl --user enable --now` failing is treated as fatal (its own
  `kdialog --error` with the systemctl output) since a daemon that isn't
  running makes the rest of the install pointless to continue.
- The self-extracting stub cleans up its temp directory in a trap on EXIT,
  so a failed or cancelled install doesn't leave extraction litter behind.

## Testing

No automated test is meaningful for a system installer that mutates
`~/.local/bin`, `~/.config/systemd/user/`, and `~/.local/share/applications/`
on the machine it runs on. Verification is manual: run the built `.run` on
this machine (safe to do — it installs to the same paths the daemon already
uses, so it's really a reinstall test) and confirm:
- Pre-flight checks pass (already-satisfied machine).
- Build succeeds, daemon installs, service is enabled and running.
- Shortcuts KCM opens and "blackout-toggle" is visible and bindable.
- `uninstall.sh` cleanly removes everything and stops the service.

## Out of scope

- Auto-installing missing distro packages (`dnf install` / `apt install`)
  — the installer names what's missing and lets the user's own package
  manager handle it.
- Any GUI beyond `kdialog` dialogs — no custom-built installer app.
- Generating or writing `kglobalshortcutsrc` entries directly — key binding
  is delegated entirely to KDE's Shortcuts settings.
- Supporting any desktop environment or compositor other than KDE
  Plasma/KWin on Wayland.
- Packaging as a distro-native format (`.rpm`/`.deb`/Flatpak) — `.run` only,
  per the request.
